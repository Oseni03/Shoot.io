## Context

The codebase has a plan system with three tiers (FREE, PRO, ENTERPRISE) defined as a Python `StrEnum` in `app/models/organization.py`. Plan limits are configured in `app/config.py` via a `PlanLimitsConfig` dataclass with one `PlanLimitEntry` per tier. Permissions are enforced through `PlanLimits.for_plan()` in `app/core/permissions.py`. Paystack billing maps plan codes via `PLAN_CODE_MAP` in `app/services/billing_service.py`. The `.env.example` has `PAYSTACK_ENTERPRISE_PLAN_CODE`. Tests for plan limits live in `tests/services/test_permissions.py`.

No shot-tracking infrastructure exists yet — no `UserMonthlyUsage` table, no `ShotService`, no usage endpoint.

## What to build

Add shot-counting to the plan system. Rename `PlanTier.ENTERPRISE` to `PlanTier.ULTIMATE`. Add `max_shots_per_month` to plan limits (FREE=3, PRO=unlimited, ULTIMATE=unlimited). Create `UserMonthlyUsage` table with `(user_id, period_start)` unique constraint — one row per user per calendar month. Build `ShotService` with `can_shoot(user)`, `record_shot(user)`, `shots_remaining(user)`. Wire `assert_shot_available()` into permissions. Serve `GET /api/v1/resumes/shots/remaining`.

A FREE user shoots 3 times then gets 402 Payment Required. PRO/ULTIMATE never hit a limit. Counter resets at calendar month start.

## Constraints

- **Additive enum migration only:** Add `ULTIMATE` to the `PlanTier` enum without removing `ENTERPRISE`. The migration adds `"ultimate"` as a valid column value but does not migrate existing rows — they stay on `"enterprise"` until their next subscription sync. This ensures zero-downtime rollback.
- **Must not be parallelised** with other migration slices (single agent on this slice).
- Follow the existing `assert_member_limit()` pattern in `app/core/permissions.py` for `assert_shot_available()`.
- `UserMonthlyUsage.period_start` is the first of the current calendar month (DATE type, not TIMESTAMP).
- User's organization plan tier is resolved via the user's primary organization membership (first org in `user.memberships`).
- Use existing `PaymentRequiredError` (402) for shot-limit responses.

## Rollback plan

1. **Enum migration:** The migration adds `ULTIMATE` without removing `ENTERPRISE`. Rollback = run the downgrade migration that drops `ULTIMATE` from the enum. Any rows with `"ultimate"` will fail the downgrade — in that case, set them to `"enterprise"` first via a data-fix script.
2. **UserMonthlyUsage table:** New table with zero data. Rollback = `DROP TABLE user_monthly_usage`.
3. **Config changes:** `max_shots_per_month` is additive — removing it is a code revert. The env var rename (`PAYSTACK_ENTERPRISE_PLAN_CODE` → `PAYSTACK_ULTIMATE_PLAN_CODE`) must be coordinated with the `.env` file update. If rolled back, old env var name must be restored.
4. **ShotService + endpoint:** Purely additive code. Rollback = revert the PR.

## Out of scope

- No web UI for shot usage display (the endpoint exists; the extension/web app will display it later).
- No metered billing or overage charging — shot counting is a hard cap, not a metered billable event.
