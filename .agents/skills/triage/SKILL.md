# Triage

Every issue passes this gate before an agent touches it. Triage is the forcing function that stops vague, under-specified, or dangerous work from entering the loop.

Every comment posted to the issue tracker during triage must start with:

```
> *This was generated by AI during triage.*
```

---

## State Labels

| Label             | Meaning                                         |
| ----------------- | ----------------------------------------------- |
| `needs-triage`    | Not yet evaluated                               |
| `needs-info`      | Waiting on reporter or requester                |
| `ready-for-agent` | Fully specified, AFK-safe                       |
| `ready-for-human` | Needs human judgment, taste, or external access |
| `wontfix`         | Will not be actioned                            |

Every issue carries exactly one state label. Flag conflicts before acting.

---

## The Gate

An issue reaches `ready-for-agent` only when all three pass.

**Scope**

- [ ] Acceptance criteria are verifiable without human eyes — a test, a CI check, or an agent assertion can confirm them
- [ ] The slice fits in one context window (~100k tokens) — split it if not
- [ ] Domain vocabulary from `DOMAIN.md` is used throughout

**Safety**

- [ ] A rollback plan exists — for migrations: reversible migration; for billing: feature flag off by default; for data: soft delete or backfill script
- [ ] Risk classification is set: `Normal` / `Migration` / `Billing` / `Isolation`
- [ ] For `Migration` or `Billing`: single-threading is confirmed — no parallel agents on this slice

**Testability**

- [ ] An integration test covers the happy path end-to-end
- [ ] Edge cases and error states appear in the acceptance criteria

If any check fails, move to `needs-info` and post the specific question that must be answered — not "please provide more info."

---

## Triaging a Specific Issue

1. **Read** the full issue — body, comments, labels, reporter, dates, prior triage notes
2. **Explore** the codebase in the area of the change. Surface relevant ADRs from `docs/adr/`
3. **Recommend** a state label and risk classification with reasoning. Wait for confirmation before acting
4. **Apply the gate.** For each failing check, post the specific question
5. **Write the agent brief** when moving to `ready-for-agent`:

```markdown
> _This was generated by AI during triage._

## Context

What the codebase looks like in the area of this change. Domain vocabulary throughout.

## What to build

The end-to-end behaviour in domain terms. No file paths.

## Constraints

- Invariants from CONTEXT.md that apply
- Relevant ADRs
- Risk classification and parallelisation rules

## Acceptance criteria

- [ ] Verifiable criterion 1
- [ ] Verifiable criterion 2

## Rollback plan

Specific steps to undo this change if it ships broken.

## Out of scope

What the agent must not touch.
```

---

## Show What Needs Attention

When invoked without a specific issue, query the tracker and show three buckets, oldest first:

1. **Unlabeled** — never triaged
2. **`needs-triage`** — evaluation in progress
3. **`needs-info` with reporter activity since last triage** — needs re-evaluation

Show counts and a one-line summary per issue. Let the maintainer pick.

---

## Quick Override

If the maintainer says "move #42 to ready-for-agent", trust them. Confirm the action (label change, comment, brief), then act. Skip the gate process. Ask whether they want an agent brief if one doesn't exist.
